BizBot Complete Documentation

An AI-Powered Business Chatbot with Natural Language to SQL
github.com/utkarsh-goel-21/chatbot

1. What Is This Project

A chatbot that lets non-technical users query their own business data using plain English. No SQL knowledge needed — the user types a question and gets a direct answer in seconds.

The system handles three types of questions through four routing pipelines:

  • TEXT_TO_SQL — Questions about live data (e.g. "How many orders did I place?") answered by generating and running a SQL query against the database.
  • RAG — Questions about trends and insights (e.g. "What are the sales trends?") answered by searching pre-generated insight documents using Retrieval-Augmented Generation.
  • HYBRID — Questions that ask for both specific data AND analysis (e.g. "How many products did I buy and what trends do you see?") runs both TEXT_TO_SQL and RAG in parallel, then combines the results into one coherent answer.
  • BLOCKED — Off-topic or conversational messages gets a polite redirect with no database or LLM pipeline run.

The system runs on the AdventureWorks dataset (a real-world bicycle sales database with 121,000+ order records) and supports two deployment modes:

ModeLLMDatabaseVector StoreUse Case
Cloud Groq API (LLaMA 3.3 70B) Supabase (hosted PostgreSQL) pgvector Production on Render + Vercel
Local Ollama (LLaMA 3.2 3B) Local PostgreSQL ChromaDB (file-based) Offline development, zero API costs

All four routing pipelines (TEXT_TO_SQL, RAG, HYBRID, BLOCKED) work identically in both cloud and local modes. The only difference is the underlying LLM and vector store. The routing logic, SQL generation, RAG retrieval, and HYBRID combination are the same codebase.

2. System Architecture

Three physical layers, one logical layer:

LayerWhat It Does
Frontend (Vercel / localhost) React + TypeScript chat interface. Manages session state with Zustand (persisted in localStorage). Sends every message with user ID and last 6 conversation turns to the backend.
Backend (Render / localhost) FastAPI Python app. Two endpoints: /chat for questions, /upload for file ingestion. All AI logic lives here. Uses either Groq API or local Ollama for LLM.
Database (Supabase / Local PG) PostgreSQL database containing both relational business data and (in cloud mode) vector embeddings. In local mode, ChromaDB handles vector storage separately.
AI Routing (logical) Lives inside the backend. Classifies each question into TEXT_TO_SQL, RAG, HYBRID, or BLOCKED before any pipeline runs.

Execution Flow

Key things to note about the flow:

  • The router always runs first, it sees the full conversation history so follow-up questions like "and on weekends?" are correctly understood in context.
  • BLOCKED returns a fixed response immediately no LLM or database call happens.
  • HYBRID runs TEXT_TO_SQL and RAG simultaneously using Python's asyncio.gather(), then makes one final LLM call to merge both answers into a single coherent response.
  • The schema fed to the SQL generator includes the actual column names and types from the database. In local mode, extra hints about table-column mappings and JOIN patterns are appended to help the smaller model.
  • RAG documents are pre-generated at server startup by running aggregation queries over the user's data and asking the LLM to write insight summaries. They are not generated per request.

3. Database Schema(AdventureWorks)

AdventureWorks is a Microsoft sample database modeled after a fictional bicycle manufacturing company called "Adventure Works Cycles". The data includes customers, orders, products, and employee information.

3.1 Schemas Overview

The database uses PostgreSQL schemas to organize tables into logical groups:

SchemaPurposeTables Used by BizBot
salesCustomer orders, transactions, sales territoriescustomer, salesorderheader, salesorderdetail
personPersonal info — names, emails, addressesperson, businessentity, emailaddress
productionProducts, categories, manufacturingproduct, productcategory, productsubcategory, productmodel

AdventureWorks has 40+ tables across these schemas, but BizBot only exposes the 10 core tables listed above to the LLM. This saves context tokens and prevents the model from getting confused by irrelevant tables like work orders or bill of materials. The other tables still exist in the database, if you want the LLM to query more tables, just add their names to the core_tables set in schema_loader.py.

4. Key Components

Backend

  • main.py — Entry point. Defines /chat and /upload endpoints. On first startup: verifies database, generates RAG documents, warms schema cache and LLM connection.
  • query_router.py — Sends question + history to LLM, gets back TEXT_TO_SQL / RAG / HYBRID / BLOCKED.
  • schema_loader.py — Fetches table names and column types from PostgreSQL. Cached in memory after first call. In Ollama mode, adds relationship hints and filters noise columns.
  • sql_generator.py — Sends schema + question to LLM. Returns raw SQL with tenant isolation. In Ollama mode, appends extra column-mapping and JOIN-pattern hints.
  • sql_executor.py — Runs the generated SQL on PostgreSQL via SQLAlchemy.
  • generate_insights.py — Runs aggregation queries per user on startup. Sends results to LLM to write insight paragraphs for RAG.
  • embedder.py — Creates 384-dim embeddings using fastembed. Stores in pgvector (cloud) or ChromaDB (local).
  • retriever.py — Embeds the user's question, runs cosine similarity search, returns top 2 matching docs.
  • groq_client.py — All LLM calls route through here. Supports Groq (with 5-key rotation) or Ollama (OpenAI-compatible API).
  • file_processor.py — Handles CSV (creates new DB table), PDF/TXT (chunked and embedded as RAG documents).

Frontend

  • chatStore.ts — Zustand store. Holds session state, auth state, theme. Persists to localStorage.
  • api.ts — sendChatMessage (POST to /chat) and uploadFile (multipart POST to /upload).

7. Why It Works This Way

  • Three pipelines + HYBRID — SQL and RAG answer structurally different question types. HYBRID handles edge cases where users ask for both data and analysis in one question, running both pipelines in parallel for speed.
  • Dual-mode architecture — The entire codebase uses runtime checks (LLM_PROVIDER, VECTOR_STORE) to switch between cloud and local. No separate branches — same codebase powers both.
  • Prompt hints only for small models — The 3B Ollama model needs explicit column-to-table mappings and JOIN patterns. The 70B Groq model doesn't. Hints are conditionally appended only when LLM_PROVIDER=ollama, keeping the production prompt lean.
  • ChromaDB instead of pgvector locally — pgvector requires a PostgreSQL extension that's complex to install on Windows. ChromaDB is pip-installable and file-based.
  • Schema caching — Fetching schema per request adds a database round-trip. Caching eliminates this after the first request.
  • Conversation history in routing — Without it, "what about on weekends?" would be misclassified. Passing the last 6 turns makes follow-ups work.

8. Complete Local Setup Guide (Step-by-Step)

You need a Windows, macOS, or Linux machine with at least 4 GB RAM. All tools used are free and open-source.

Clone the Repository

git clone https://github.com/utkarsh-goel-21/chatbot.git
cd chatbot

This creates a chatbot/ folder with the full project.

Install PostgreSQL

Download and install PostgreSQL from postgresql.org/download.

  • Windows: Use the EDB installer. During setup, set the superuser password to password (or any password you'll remember). Keep the default port 5432. Uncheck "Stack Builder" at the end.
  • macOS: brew install postgresql@16 then brew services start postgresql@16
  • Linux: sudo apt install postgresql postgresql-contrib

Verify it's running:

PowerShell (Windows)Get-Service postgresql*
# Should show Status: Running
Common Issue: "psql is not recognized" On Windows, psql is NOT in PATH by default. Use the full path: "C:\Program Files\PostgreSQL\18\bin\psql.exe" (adjust the version number to match your install — it might be 16 or 17).

Create the Database and Load Data

First, create the adventureworks database:

PowerShell (Windows)$env:PGPASSWORD='password'
& "C:\Program Files\PostgreSQL\18\bin\psql.exe" -U postgres -c "CREATE DATABASE adventureworks;"

Then load the AdventureWorks data:

PowerShell (Windows)$env:PGPASSWORD='password'
& "C:\Program Files\PostgreSQL\18\bin\psql.exe" -U postgres -d adventureworks --no-psqlrc --pset pager=off --set ON_ERROR_STOP=off -f aw_data_only.sql
This takes 3–5 minutes The file is ~78MB with hundreds of thousands of INSERT statements. You'll see many lines scrolling by — this is normal. Some ERROR messages about duplicate keys or missing schemas are expected and safe to ignore.

Verify the data loaded correctly:

PowerShell (Windows)$env:PGPASSWORD='password'
& "C:\Program Files\PostgreSQL\18\bin\psql.exe" -U postgres -d adventureworks -c "SELECT COUNT(*) FROM sales.customer;"

Expected result: 19820. If you see this, the data is loaded correctly.

Common Issue: "FATAL: password authentication failed" If you set a different password during PostgreSQL installation, use that password instead of password everywhere. Also update it in the .env file (Step 5).

Install Ollama and Pull the Model

Important: Ollama version must be 0.18.3 or lower Ollama v0.19.0 has a confirmed bug that crashes llama3.2:3b with llama runner process has terminated. To avoid this, download v0.18.3 directly:
  1. Go to github.com/ollama/ollama/releases/tag/v0.18.3
  2. Scroll down to Assets and download OllamaSetup.exe (Windows) or the appropriate installer for your OS
  3. Run the installer — it installs and starts Ollama automatically
  4. Verify the version: ollama --version → should say 0.18.3

After installation, pull the LLaMA 3.2 model:

ollama pull llama3.2:3b

This downloads a ~2GB model file. It only happens once.

Test that it works:

ollama run llama3.2:3b "say hi"

If it responds with text, you're good. If it sits for more than 30 seconds, you may need to free up RAM.

Common Issue: "model requires more system memory" The model needs ~1.7 GB free RAM. Close other apps (Chrome, VS Code, etc.) and retry. Check Task Manager → Processes to see what's consuming memory.

Configure the Backend Environment

Copy the example config and set it to local mode:

PowerShellcd chatbot
Copy-Item .env.example .env

Open .env in any text editor and replace its entire contents with:

# BizBot Local Mode Configuration
LLM_PROVIDER=ollama
OLLAMA_BASE_URL=http://localhost:11434/v1
OLLAMA_MODEL=llama3.2:3b

DATABASE_URL=postgresql://postgres:password@localhost:5432/adventureworks

VECTOR_STORE=chromadb
Replace password with YOUR PostgreSQL password If you set a different password during PostgreSQL installation, change it in the DATABASE_URL line.

Set Up the Python Backend

Create a virtual environment and install dependencies:

PowerShell (Windows)cd chatbot
python -m venv venv
.\venv\Scripts\activate
pip install -r requirements.txt
macOS / Linuxcd chatbot
python3 -m venv venv
source venv/bin/activate
pip install -r requirements.txt
First install takes 2–5 minutes This installs FastAPI, SQLAlchemy, ChromaDB, fastembed, and other dependencies. ChromaDB and fastembed are the largest packages.
Common Issue: "Microsoft Visual C++ 14.0 or greater is required" Some packages need C++ build tools on Windows. Install them from Visual Studio Build Tools — select "Desktop development with C++" workload, then retry pip install -r requirements.txt.

Set Up the Frontend

Install Node.js (v18+) from nodejs.org if you don't have it.

cd frontend
npm install

Create a file called .env inside the frontend/ folder with this content:

VITE_SUPABASE_URL=https://placeholder.supabase.co
VITE_SUPABASE_ANON_KEY=placeholder_key

The frontend code imports Supabase for auth in cloud mode. In local mode, auth is skipped but the import still runs and expects these variables to exist. Placeholder values prevent the initialization error — they won't make any API calls.

Start the Backend

Make sure Ollama is running (check system tray or run ollama list), then:

Terminal 1cd chatbot
.\venv\Scripts\activate
uvicorn main:app --reload --port 8000

Wait until you see:

🚀 All systems warmed up and ready!
First boot takes 1–3 minutes It generates AI insight documents for the sample users and downloads the embedding model (~91 MB, one-time). Subsequent boots are much faster. You'll see some SAWarning: Did not recognize type 'xml' messages — these are harmless and can be ignored.

Leave this terminal open.

Start the Frontend

Terminal 2cd chatbot/frontend
npm run dev

Wait until you see:

VITE v7.x.x  ready in XXX ms
  ➜  Local:   http://localhost:5173/

Test It!

Open your browser and go to http://localhost:5173

Try these questions:

  • "Hello" → Should respond with a greeting (BLOCKED pipeline)
  • "What is my name?" → Should return the customer's name (TEXT_TO_SQL)
  • "How many orders do I have?" → Should count orders (TEXT_TO_SQL)
  • "What are the sales trends?" → Should discuss trends (RAG)
  • "How many products did I buy and what trends do you see?" → Data + insights (HYBRID)

10. Shutting Down & Restarting

Shutting Down

  1. Frontend: Press Ctrl+C in the frontend terminal
  2. Backend: Press Ctrl+C in the backend terminal
  3. Ollama: Right-click the system tray icon → Quit (or taskkill /IM ollama.exe /F)
  4. PostgreSQL: Runs as a service — no need to stop it (auto-starts on boot)

Restarting (After Reboot)

PostgreSQL and Ollama auto-start on boot. You just need to:

  1. Open terminal → cd chatbot → .\venv\Scripts\activate → uvicorn main:app --reload --port 8000
  2. Open second terminal → cd chatbot/frontend → npm run dev
  3. Open browser → http://localhost:5173
